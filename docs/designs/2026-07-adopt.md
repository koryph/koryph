# koryph adopt: the idiot-proof front door for existing repos (2026-07-21)

Status: ACCEPTED — implementation beads filed under the adopt epic.
Companion: docs/designs/2026-07-scaffolding.md (koryphization, §3–4),
docs/designs/2026-07-software-factory.md (§3.1 `koryph new`).

## 1. Why this is the most important feature

Adoption is the product's funnel. A user who `brew install`s koryph today
faces an eight-step, four-document journey: install `claude` and `bd` by
hand, run `bd init` by hand (the docs' one hard prerequisite), type
`koryph project add` with two required flags they must already understand
(`--account`, `--identity`), then hand-edit `koryph.project.json` (`gate`,
`area_map`, `forge` — llms.txt: "cannot be guessed"), then `validate`,
then discover signing/posture/ci/release as separate tracks. Every one of
those steps is a place to give up.

Measured against the goal — *a user can go from `brew install koryph` to a
green `koryph validate` in one command and a handful of confirmations* —
today's flow fails concretely:

- `koryph project add` on a fresh repo leaves `beads_status: "none"` and
  `koryph validate` fails with raw `bd` stderr ("no beads database found…
  run 'bd init'"). koryph never initializes, migrates, or hardens a beads
  DB; it only probes one.
- `koryph init` verifies `git`/`claude`/`bd` but never offers to install
  what is missing — on any platform. The Homebrew cask declares no
  dependencies. The consumer-nix story is one doctor remediation.
- Nothing derives the account identity, though `~/.claude.json` (and each
  `CLAUDE_CONFIG_DIR`'s `.claude.json`) knows it and
  `internal/account.Verify` already reads it.
- `gate`/`area_map`/`forge` are declared unguessable, but a Makefile, a
  `package.json`, a `go.mod`, and `git remote get-url origin` guess all
  three well enough to *propose and confirm*.
- Paper cuts that break trust in the first five minutes: `koryph init`
  does not create `~/.koryph/slots`, so `koryph doctor` reports an error
  immediately after a clean init; running `bd init` after `project add`
  double-registers session priming (koryph-prime.sh *and* bd's own
  `bd prime --hook-json` SessionStart hook); `bd init` auto-commits its
  files while koryph's assets sit untracked, so the repo lands in a
  half-committed state no wizard ever reconciles.

## 2. The verb

```
koryph adopt [<root>] [flags]
```

One command that takes any existing git repo to a green `koryph validate`,
or tells you precisely what is missing and offers to fix it. It is the
**koryphization tail** from docs/designs/2026-07-scaffolding.md §4,
promoted to a standalone, wizard-shaped front door. `koryph new`
(koryph-om7, on hold) stays the greenfield verb and will call this same
tail after scaffolding; nothing here decomposes or blocks that epic.

`adopt` replaces no existing verb. `project add`, `install-assets`,
`validate`, `doctor` keep their contracts; `adopt` sequences them and
fills the gaps between them. Re-running `adopt` on a healthy project is a
no-op report (it doubles as an onboarding-scoped doctor).

## 3. Shape: detect → plan → confirm → execute → verify

Five phases, strictly ordered. All detection is read-only; nothing writes
before the plan is confirmed.

### 3.1 Detect (read-only)

| Probe | Source | Feeds |
|---|---|---|
| platform + package manager | GOOS, /etc/os-release, `brew`/`apt-get`/`dnf`/`pacman`/`zypper`/`nix` on PATH | dependency install plan |
| repo nix posture | `flake.nix` + `.envrc` + `direnv` on PATH | flake-first dependency route |
| binaries | `git`, `claude`, `bd` (+ version ≥ 1.0.5), `gh` (optional) | dependency install plan |
| runtime auth | `internal/account.Verify` on candidate profiles | account step |
| account candidates | `~/.claude.json`; `~/.claude-*/.claude.json`; `.envrc` claude-account managed block (already parsed by `onboard.Inspect`) | account step |
| git facts | remote URL → forge (github/gitlab), default branch, dirty tree | register + config |
| beads state | `.beads/` present, `bd doctor`, version, hardened (issues.jsonl ignored + `sync.remote`), git hooks | beads step |
| gate candidates | Makefile targets (`gate`>`test`/`lint`/`build`), package.json scripts, go.mod, Cargo.toml, pyproject | `gate` proposal |
| layout | top-level source dirs (minus dot-dirs, docs, vendor, node_modules) | `area_map` proposal |
| koryph state | registry record, `koryph.project.json`, asset drift, settings wiring | skip-done marking |

### 3.2 Plan

The plan is a printed, ordered list of steps, each carrying **what** it
will do, **why** (one sentence, plain language), **what it writes**, and a
state: `done` (skip), `needed`, `offer` (optional, default-off), or
`blocked` (with the unblocking action). Example:

```
ADOPTION PLAN — myrepo (/Users/me/src/myrepo)

  done    tools        git 2.49, claude 2.1 (authed), gh 2.62
  needed  tools        bd not found → install via Homebrew (brew install gastownhall/beads/bd)
  done    home         ~/.koryph initialized
  needed  beads        initialize issue DB (bd init --prefix myrepo --remote git+https://github.com/me/myrepo.git)
                       why: koryph dispatches work from the beads ready-graph; without it the loop has nothing to build
  needed  register     account personal <me@example.com> (derived from ~/.claude.json; confirm)
  needed  config       gate: make test, make lint (detected from Makefile; confirm)
                       forge: github (from origin remote)
                       area_map: 4 areas proposed from src/, cmd/, docs/, web/
  needed  assets       AGENTS.md, 14 agents, 10 commands, hooks + settings.json
  offer   signing      vault-backed commit signing (koryph signing keygen for the no-vault path)
  offer   posture      GitHub hygiene profile oss-solo-maintainer
  needed  commit       one signed commit of the koryph/beads assets on a branch you approve
  needed  verify       koryph validate must pass green
```

### 3.3 Confirm

Three consent scopes, never mixed into one blanket yes:

1. **Repo scope** — everything written inside the repo and `~/.koryph`.
   One consolidated y/N for the plan as printed.
2. **System scope** — each package-manager install and each `flake.nix`
   edit is confirmed individually, showing the exact command/diff before
   it runs. Never sudo without showing the command. Declining a system
   change downgrades that step to `blocked` with manual instructions and
   the wizard continues where it can.
3. **Value confirmations** — derived values the user must own: account +
   identity, gate commands, forge. Presented with their provenance
   ("derived from ~/.claude.json") and a one-keystroke accept. The gate
   is always confirmed explicitly: a wrong gate green-lights garbage.

Interactive prompts follow the established idiom (stderr prompts,
`isTerminal` guard, stdout reserved for machine output). Non-TTY or
`--yes`: derived values are accepted only where provenance is
unambiguous; anything ambiguous fails closed with the flag to set
(`--account`, `--identity`, `--gate`, …). `--dry-run` stops after the
plan. `--json` emits plan and results machine-readably (agent-drivable —
this is the llms.txt runbook collapsed into one command).

### 3.4 Execute

Dependency-ordered, each step idempotent, streamed `ok/skip/fail` lines:

1. **deps** — install missing tools by the platform route (§4).
2. **home** — `koryph init` layout, *including* `slots/`, `slots/demand/`,
   `telemetry/` (doctor parity, §7).
3. **beads** — init or migrate + harden (§5). Runs BEFORE asset install
   so bd's settings/AGENTS.md integration lands first and koryph's
   settings merge migrates bd's prime hook in place (no duplicates).
4. **register + config** — `onboard.Register` with derived account, then
   write confirmed `gate`/`forge`/`area_map` into `koryph.project.json`
   (§6).
5. **assets** — the existing `project add` asset sequence (AGENTS.md,
   personas, commands, rules/settings merge with prime dedupe).
6. **offers** — signing, posture: explained, default-off, one keystroke
   to accept; declining prints the later command.
7. **commit** — offer one signed conventional commit ("chore: adopt
   koryph") of everything the wizard wrote, so the repo never lands
   half-committed. Declining leaves the tree dirty with a summary.
8. **verify** — run the `koryph validate` check sequence in-process;
   print next steps (`/koryph-plan`, `koryph run --once --dry-run`).

### 3.5 Verify

`adopt` ends with the validate gate green or with a precise, remediated
failure list. Exit 0 only on green (or `--dry-run`).

## 4. Dependencies: detect the platform, ask, install

New package `internal/sysdeps`:

- **Detection**: GOOS; on linux, `/etc/os-release` ID/ID_LIKE
  (debian/ubuntu, fedora/rhel, arch, opensuse); package managers on PATH.
- **Route order**: repo flake (§4.1, when `flake.nix` exists and `nix` is
  on PATH) → Homebrew (darwin, linuxbrew) → native (apt/dnf/pacman/
  zypper) → `nix profile install` → manual instructions (URL + command).
- **Tool specs** (data, not code): `bd` (brew `gastownhall/beads/bd`; else
  release-tarball instructions), `claude` (brew cask / `npm i -g
  @anthropic-ai/claude-code` / native instructions), `gh` (ubiquitous).
  Specs carry a post-install verify command (`bd version`, `claude
  --version`).
- **Consent**: every install prints the exact argv and asks (system
  scope, §3.3). No curl-pipe-sh, ever. Failure → `blocked` + manual
  instructions; the wizard continues.

### 4.1 Nix-first repos

When the target repo has `flake.nix`: offer to add the beads flake input
and `bd` to the devShell packages instead of a system install. The edit
is minimal and structural (insert input + package entry), shown as a
unified diff before writing, followed by `nix flake lock` and a
`direnv exec <root> bd version` verification. `internal/beads/version.go`
already knows how to find the beads input for the doctor's upgrade
remedy; the same knowledge drives insertion. If the flake edit is
declined or fails, fall back to the system route with consent.

## 5. Beads: initialize, migrate, harden — through bd, never around it

koryph keeps its one-adapter rule: everything goes through the `bd` CLI
(`internal/beads`), never raw Dolt. New adapter surface:

- **`EnsureDB(root, opts)`** — absent DB: `bd init --non-interactive
  --init-if-missing --prefix <project-id> [--remote <derived>]`. The
  sync remote is derived from the git origin URL (`git+https://…` form,
  as this repo's own `.beads/config.yaml` does) and confirmed with the
  user; no remote → local-only init plus a printed note about
  `bd dolt push` when a remote appears.
- **Existing DB — migrate**: run `bd doctor` (it owns schema/version
  migration checks: DB version, migration status, schema compatibility,
  hooks, .gitignore currency); surface its findings as adopt steps;
  `Adapter.Snapshot()` (tar backup, already implemented) before any
  fixing operation bd performs. Version gate: `beads.ProbeVersion() ≥
  MinVersion` with the existing `Remediation()` text routed through the
  §4 installer instead of prose.
- **Harden**: the `onboard.Inspect` hardened-check made writable —
  ensure `.beads/.gitignore` covers `issues.jsonl`, ensure `sync.remote`
  set (with consent), ensure bd git hooks installed.
- **What koryph never does**: own bd's schema, write Dolt, or fork bd's
  migration logic. "Migrating an existing bead DB to match the schema
  koryph uses" resolves to: bd's own migration (via `bd doctor`/upgrade)
  + koryph's *conventions* layered as data (labels, `area_map`) — §6.

## 6. Config inference: propose, confirm, write

The three "cannot be guessed" fields become "guessed, confirmed, written":

- **forge** — remote host match (github.com → `github`, gitlab →
  `gitlab`, else ask/omit).
- **gate** — ranked candidates: `make gate` if the target exists, else
  Makefile `test`/`lint`/`build` targets, `package.json` scripts,
  `go vet ./...` + `go test ./...` for go.mod, `cargo test`, pytest.
  Presented as a checklist, always explicitly confirmed (§3.3). The
  wizard offers to *run the gate once* before accepting it, so the user
  learns immediately whether their baseline is green.
- **area_map** — top-level source dirs → `<dir>: [<lang>:<dir>]`
  proposals, presented as the starter map with one line of explanation
  (footprints are how parallel agents avoid merge conflicts). Accepting
  a rough map is fine; the fence is honesty, not completeness —
  `/koryph-plan` refines per-bead labels later.

Inference lives in `internal/onboard` (beside `Inspect`), pure and
unit-testable: `InferGate(root)`, `InferForge(remoteURL)`,
`InferAreaMap(root)` returning proposals with provenance strings.

## 7. Paper cuts fixed alongside (trust is lost in the first five minutes)

1. `registry.Store.Init` creates `slots/`, `slots/demand/`, `telemetry/`
   — `koryph doctor` must be clean immediately after `koryph init`.
2. `rules.ensureHook` dedupes: when the koryph-prime.sh entry is present,
   any *additional* SessionStart entry whose command is bare
   `bd prime --hook-json` is removed (bd-init-after-project-add
   double-prime). `doctor --project` gains a duplicate-prime finding.
3. `onboard.Validate`'s beads failure becomes a koryph-voiced remediation
   ("no beads database — run `koryph adopt` (or `bd init`)") instead of
   raw bd stderr.
4. docs/user-guide/projects-and-accounts.md:15 "Created by `koryph
   onboard`" corrected to `koryph project add` / `koryph adopt`.
5. `koryph init` next-steps text and `help getting-started` lead with
   `koryph adopt`.

## 8. Docs

New chapter docs/user-guide/adopt.md (the wizard, each step's why, the

## 9. Non-goals (the fence)

- No vault account provisioning (Proton Pass/1Password setup stays the
  user's; `signing keygen` is the offered no-vault path).
- No GitHub App/bot provisioning inside adopt (signposted next step).
- No app-layout opinions in `area_map` inference beyond naming what
  exists (scaffolding fence, docs/designs/2026-07-scaffolding.md §1).
- No `koryph new` decomposition — koryph-om7 unchanged; adopt is the
  shared tail it will call.
- No background/daemon anything; adopt is one foreground run.

## 10. Acceptance

From a fresh macOS or Ubuntu machine with only git + brew/apt and an
authenticated `claude`:

```
brew install koryph/tap/koryph
cd ~/src/myrepo
koryph adopt
```

…reaches `koryph validate` green with: ≤ 1 consolidated repo consent,
per-item system consents, ≤ 3 value confirmations (account, gate,
forge/area accepted together), zero hand-edited files, zero raw bd
errors, a clean `koryph doctor`, and one offered signed commit. Re-run
prints an all-`done` plan and exits 0.
